# Design: Personal Korean Stock Auto-Trading Dashboard

## Problem Statement

A code-literate trader is managing Korean stock trading strategies through fragmented Python scripts. The workflow from signal detection to order execution requires multiple disconnected tools, manual steps, and context switching between scripts, Korean brokerage HTS tools, and spreadsheets. The HTS tools (키움증권, 한국투자증권 etc.) have the market data but offer terrible UX and zero customizability. The result is a slow, error-prone, and frustrating workflow that defeats the purpose of having automated strategies.

## What Makes This Cool

A single web dashboard that replaces an entire HTS application + scattered Python scripts + manual monitoring. You open one browser tab in the morning, your strategies are already running, and you can see every signal, every position, every trade — with the ability to intervene or let the system run autonomously. It's a personal command center for the Korean stock market, built exactly the way you want it because you own every line of code.

## Constraints

- **Korean market hours:** KRX operates 09:00-15:30 KST. The system needs to be running and connected during these hours.
- **API rate limits:** 한국투자증권 Open API limits REST calls to ~20 requests/second and WebSocket subscriptions to 40 concurrent streams. Architecture must respect these limits with a token-bucket rate limiter.
- **Brokerage API dependency:** Must integrate with 한국투자증권 Open API (or 키움증권 OpenAPI). These APIs require registered accounts and API keys.
- **Personal use only:** No multi-tenancy, no auth system, no onboarding flow needed. Runs on localhost or a personal server.
- **Existing Python scripts:** The trading strategy logic already exists as Python code. The system wraps around it, not replaces it.

## Premises

1. Web dashboard (not CLI/mobile) — primary interaction is monitoring from a desk during Korean market hours.
2. Core value = unifying existing Python strategy scripts with real-time market data and order execution — not building a new strategy framework.
3. Korean stock market (KRX) is the primary market — using 한국투자증권 Open API for data and execution.
4. "Full automation" = autonomous execution WITH monitoring dashboard for human oversight and intervention — not unattended fire-and-forget.
5. Personal tool — prioritize development speed and customizability over polish.

## Recommended Approach: Python Full Stack

FastAPI backend + React/Vite frontend (revised from Next.js per eng review).

### Key Components

1. **Broker Adapter** (`app/services/broker_adapter.py`) — Thin layer over python-kis:
   - REST client: Orders, account info, historical data. Token-bucket rate limiter (~20 req/s).
   - WebSocket client: Real-time price streaming. Max 40 concurrent subscriptions.
   - SubscriptionManager: Dict-based ticket storage, prevents GC-based silent unsub.

2. **Strategy Engine** (`app/services/strategy.py`) — Loads user strategy scripts via importlib:
   ```python
   MarketContext = {"ticker": str, "ohlcv": list[OHLCV], "portfolio": PortfolioState, "timestamp": datetime}
   Signal = {"action": "BUY"|"SELL"|"HOLD", "ticker": str, "confidence": float, "reason": str}
   ```
   Runs on asyncio tasks during market hours (09:00-15:30 KST).

3. **Order Manager** (`app/services/orders.py`) — Risk controls:
   - Max position size: configurable (default 10% of portfolio per ticker)
   - Stop-loss: configurable (default 3%)
   - Daily loss limit: configurable (default 5% of portfolio)
   - Auto/manual mode with emergency stop

4. **WebSocket Hub** (`app/ws/hub.py`) — Real-time data streaming to frontend.

5. **Dashboard UI** (`frontend/`) — Four-panel layout:
   - Chart (top-left, 2/3 width) with stock tabs
   - Portfolio summary (top-right, 1/3 width)
   - Strategy signals (bottom-left) with execute/ignore
   - Trade log (bottom-right) with chronological entries

### Database Schema

```sql
CREATE TABLE trades (
  id INTEGER PRIMARY KEY,
  ticker TEXT NOT NULL,
  action TEXT NOT NULL,
  quantity INTEGER NOT NULL,
  price REAL NOT NULL,
  broker_order_id TEXT,
  strategy_name TEXT,
  signal_confidence REAL,
  status TEXT DEFAULT 'PENDING',
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE signals (
  id INTEGER PRIMARY KEY,
  ticker TEXT NOT NULL,
  action TEXT NOT NULL,
  confidence REAL,
  reason TEXT,
  strategy_name TEXT,
  acted_on INTEGER DEFAULT 0,
  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE positions (
  ticker TEXT PRIMARY KEY,
  quantity INTEGER NOT NULL,
  avg_price REAL NOT NULL,
  current_price REAL,
  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```
